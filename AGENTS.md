# エージェント指示書

## コア原則

1. **レスポンス** - 必ず日本語で回答してください
2. **エージェントファースト** — ドメインタスクは専門エージェントに委譲する
3. **テスト駆動** — 実装前にテストを書く。カバレッジ80%以上を必須とする
4. **セキュリティファースト** — セキュリティを妥協しない。すべての入力を検証する
5. **イミュータビリティ** — 常に新しいオブジェクトを生成し、既存のオブジェクトを変更しない
6. **実行前に計画する** — 複雑な機能はコードを書く前に計画を立てる

## 利用可能なエージェント

| エージェント | 目的 | 使用タイミング |
|-------------|------|--------------|
| planner | 実装計画の立案 | 複雑な機能追加・リファクタリング |
| tdd-guide | テスト駆動開発 | 新機能追加・バグ修正 |
| code-reviewer | コード品質と保守性 | コードを書いた・修正した直後 |
| security-reviewer | セキュリティレビュー | コミット前 |

## エージェントオーケストレーション

ユーザーの指示を待たず、以下の状況では自律的にエージェントを起動すること:
- 複雑な機能要求 → **planner**
- コードを書いた・修正した直後 → **code-reviewer**
- バグ修正・新機能追加 → **tdd-guide**
- セキュリティ関連のコード → **security-reviewer**

独立した操作には並列実行を使用し、複数のエージェントを同時に起動すること。

## セキュリティガイドライン

**すべてのコミット前に確認すること:**
- シークレットのハードコードがない（APIキー・パスワード・トークン）
- すべてのユーザー入力がバリデーションされている
- SQLインジェクション対策がされている（パラメータ化クエリを使用）
- XSS対策がされている（HTMLがサニタイズされている）
- CSRF対策が有効になっている
- 認証・認可が正しく実装されている
- すべてのエンドポイントにレートリミットが設定されている
- エラーメッセージが機密情報を漏洩していない

**シークレット管理:** ソースコードにシークレットをハードコードしない。環境変数またはシークレットマネージャーを使用する。起動時に必要なシークレットの存在を確認する。漏洩したシークレットは即座にローテーションする。

**セキュリティ問題を発見した場合:** 即座に停止 → security-reviewer エージェントを使用 → CRITICAL の問題を修正 → 漏洩シークレットをローテーション → コードベース全体で同様の問題を確認する。

## コーディングスタイル

**イミュータビリティ（重要）:** 常に新しいオブジェクトを生成し、既存のオブジェクトを変更しない。変更を加えた新しいコピーを返す。

**ファイル構成:** 少数の大きなファイルより多数の小さなファイルを優先する。目安は200〜400行、最大800行。機能・ドメイン単位で整理し、高凝集・疎結合を意識する。

**エラー処理:** 各レイヤーでエラーを処理する。UIコードではユーザーフレンドリーなメッセージを提供する。サーバーサイドでは詳細なコンテキストをログに記録する。エラーを握りつぶさない。

**入力バリデーション:** システム境界ですべてのユーザー入力を検証する。スキーマベースのバリデーションを使用する。明確なメッセージでフェイルファストを実践する。外部データを無条件に信頼しない。

**コード品質チェックリスト:**
- 関数は小さく（50行未満）、ファイルは適切な規模に収める（800行未満）
- ネストが深すぎない（4段階以下）
- 適切なエラー処理、ハードコードされた値がない
- 読みやすく、適切な命名がされている

## テスト要件

**最低カバレッジ: 80%**

テスト種別（すべて必須）:
1. **ユニットテスト** — 個々の関数・ユーティリティ・コンポーネント
2. **インテグレーションテスト** — APIエンドポイント・DB操作
3. **E2Eテスト** — 重要なユーザーフロー

**TDDワークフロー（必須）:**
1. 先にテストを書く（RED） — テストが失敗することを確認する
2. 最小限の実装を行う（GREEN） — テストが通ることを確認する
3. リファクタリングする（IMPROVE） — カバレッジ80%以上を確認する

テスト失敗時: テストの独立性を確認 → モックを検証 → 実装を修正する（テスト自体が誤っている場合を除く）。

## 開発ワークフロー

1. **計画** — planner エージェントを使用し、依存関係とリスクを洗い出し、フェーズに分割する
2. **TDD** — tdd-guide エージェントを使用し、テストを先に書いて実装・リファクタリングする
3. **レビュー** — コードを書いたら即座に code-reviewer エージェントを使用し、CRITICAL/HIGH の指摘に対応する
4. **知識を適切な場所に記録する**
   - 個人的なデバッグメモ・好み・一時的なコンテキスト → 自動メモリ
   - チーム・プロジェクトの知識（アーキテクチャの意思決定・API変更・ランブック） → プロジェクトの既存ドキュメント構造
   - 現在のタスクで関連ドキュメントやコードコメントがすでに生成される場合、同じ情報を重複して記録しない
   - 適切なプロジェクトドキュメントの置き場所が不明な場合、新しいトップレベルファイルを作成する前に確認する
5. **コミット** — Conventional Commits 形式に従い、包括的な PR サマリーを記述する

## Git ワークフロー

**コミット形式:** `<type>: <description>` — type 一覧: feat, fix, refactor, docs, test, chore, perf, ci

**PR ワークフロー:** コミット履歴全体を分析 → 包括的なサマリーを作成 → テスト計画を記載 → `-u` フラグを付けてプッシュする。

## アーキテクチャパターン

**API レスポンス形式:** 成功インジケーター・データペイロード・エラーメッセージ・ページネーションメタデータを含む統一エンベロープ形式を使用する。

**リポジトリパターン:** 標準インターフェース（findAll, findById, create, update, delete）の背後にデータアクセスを隠蔽する。ビジネスロジックはストレージの実装ではなく抽象インターフェースに依存する。

**スケルトンプロジェクト:** 実績のあるテンプレートを探し、並列エージェントで評価（セキュリティ・拡張性・適合度）し、最適なものをクローンして実証済みの構造の中でイテレーションする。

## パフォーマンス

**コンテキスト管理:** 大規模なリファクタリングやマルチファイル機能ではコンテキストウィンドウの残り20%を使い切らないようにする。単一編集・ドキュメント・単純な修正など感度の低いタスクは高い利用率でも許容される。

## 完了基準

- すべてのテストがカバレッジ80%以上で通過している
- セキュリティ脆弱性がない
- コードが読みやすく保守性が高い
- パフォーマンスが許容範囲内である
- ユーザーの要件がすべて満たされている
