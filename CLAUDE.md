# CLAUDE.md

よくあるLLMのコーディングミスを減らすための行動ガイドライン。必要に応じてプロジェクト固有の指示と統合してください。

**トレードオフ:** これらのガイドラインは、スピードよりも慎重さを重視しています。些細なタスクについては、適切に判断して進めてください。

## 1. コーディングの前に考える

**推測しない。混乱を隠さない。トレードオフを表面化させる。**

実装の前に:
- 自分の前提条件を明確に述べること。不確実な場合は質問する。
- 複数の解釈が存在する場合は、それらを提示すること。黙って1つを選ばない。
- よりシンプルなアプローチが存在する場合は、そう伝えること。正当な理由がある場合は反論（プッシュバック）する。
- 不明な点があれば立ち止まること。何が混乱を招いているかを明確にし、質問する。

## 2. シンプルさを最優先する

**問題を解決する最小限のコードに留める。推測に基づいたものは一切含めない。**

- 要求された以上の機能は追加しない。
- 一度きりしか使わないコードの抽象化は行わない。
- 要求されていない「柔軟性」や「設定可能性（コンフィギュラビリティ）」は持たせない。
- 起こり得ないシナリオに対するエラーハンドリングは行わない。
- 50行で済むところを200行で書いてしまった場合は、書き直すこと。

自問してください：「シニアエンジニアはこれを複雑すぎると言うだろうか？」。もしそうなら、シンプルにしてください。

## 3. 局所的で精密な変更 (Surgical Changes)

**必要な箇所にのみ触れること。自分が散らかしたものだけを片付ける。**

既存のコードを編集する場合:
- 隣接するコード、コメント、フォーマットを「改善」しない。
- 壊れていないものをリファクタリングしない。
- 自分のやり方とは異なっていたとしても、既存のスタイルに合わせる。
- 無関係なデッドコードに気づいた場合は、言及するにとどめ、削除はしない。

自分の変更によって孤立したコード（不要になったもの）が生じた場合:
- **自分**の変更によって使われなくなったインポート、変数、関数を削除する。
- 指示されない限り、以前から存在するデッドコードは削除しない。

判断基準: 変更されたすべての行は、ユーザーのリクエストに直接結びついている必要があります。

## 4. 目標主導の実行

**成功基準を定義する。検証されるまでループする。**

タスクを検証可能な目標に変換する:
- 「バリデーションを追加する」 → 「無効な入力に対するテストを書き、それをパスさせる」
- 「バグを修正する」 → 「バグを再現するテストを書き、それをパスさせる」
- 「Xをリファクタリングする」 → 「リファクタリングの前後でテストがパスすることを確認する」

複数のステップにわたるタスクの場合は、簡単な計画を提示する:
```
1. [ステップ] → 検証: [チェック内容]
2. [ステップ] → 検証: [チェック内容]
3. [ステップ] → 検証: [チェック内容]
```

強力な成功基準があれば、自律的にループを回すことができます。弱い基準（「とりあえず動くようにする」など）では、絶えず確認が必要になってしまいます。

---

**これらのガイドラインが機能している証拠:** diff（差分）における不要な変更が減り、複雑化しすぎたことによる書き直しが減り、ミスをした後ではなく実装前に明確化のための質問がなされること。
