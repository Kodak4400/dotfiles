# GitHub Pull Request 作成

ブランチの変更履歴全体を分析し、レビュアーが理解しやすい構造化されたPR説明文を生成する。ユーザーの承認を得てからPRを作成する。

## 主要な概念

- **承認ファースト**: ユーザーの承認なしに `gh pr create` を実行してはならない
- **全履歴を分析**: 最新コミットだけでなく、ベースブランチからの差分全体を確認する
- **「なぜ」を伝える**: コードから読み取れない背景・意図・判断を必ず補足する
- **チケット連携**: ブランチ名からチケット番号を抽出し、`Closes #番号` 形式で記載する

## 使用タイミング

以下の場合に起動する:

- 「PRを作って」「プルリクを出して」などPR作成を求められた場合
- コミット・プッシュが完了し、レビュー依頼の準備ができた場合
- PR説明文の草案・提案を求められた場合

## 使い方

### ステップ1: ブランチ情報と変更履歴の収集

以下のコマンドを並列で実行し、PR対象の全変更を把握する:

- `git branch --show-current` — 現在のブランチ名（ソースブランチ）を取得する
- `git log main...HEAD --oneline` — ベースブランチからのコミット一覧を取得する
- `git diff main...HEAD --stat` — 変更ファイルの一覧と行数を取得する
- `git diff main...HEAD` — 変更の詳細を取得する
- `gh pr list --state open --limit 5` — 既存PRのタイトルスタイルを参考にする

> ベースブランチが `main` 以外（`develop`、`master` など）の場合は適宜変更する。

### ステップ2: チケット番号の抽出

ブランチ名からチケット番号を抽出し、`Closes #番号` 形式でPRに含める:

- `feature/123-login-improvement` → `Closes #123`
- `bugfix/456-validation-error` → `Fixes #456`
- `refactor/789-code-cleanup` → `Relates to #789`

ブランチ名にチケット番号が含まれない場合、またはクローズ対象のIssueが不明な場合はユーザーに確認する。

### ステップ3: PR説明文の生成

以下の構造でPR説明文を生成する:

```markdown
## Summary
<!-- このPRで何を・なぜ変更したかを2〜3行で述べる -->

## Changes
<!-- 変更内容を機能・ファイル単位で箇条書きにする -->
- `src/foo.ts`: 〜のため〜を修正
- `src/bar.ts`: 〜のため〜を追加

## Context
<!-- コードから読み取れない背景・設計判断・制約を記述する -->
- なぜこの実装アプローチを選択したか
- 検討した代替案とその不採用理由
- 関連するドキュメント・仕様・設計書へのリンク

## Testing
<!-- レビュアーが動作確認するための手順と、実施済みテスト内容 -->
- [ ] ○○の操作で△△が正しく動作することを確認
- [ ] 既存テストが通ることを確認

## Impact / Risks
<!-- この変更による影響範囲と注意事項 -->
- 影響を受けるコンポーネント・機能
- 破壊的変更・マイグレーション・環境変数の変更有無

## Related Issues
Closes #番号
```

**各セクションの記述ルール:**

| セクション | 目的 | 記述のポイント |
|---|---|---|
| Summary | 変更の全体像 | 「〜のため〜を修正/実装した」の構成で1〜3行 |
| Changes | 具体的な変更内容 | ファイル名を含む箇条書き。「何を」「なぜ」をセットで書く |
| Context | 設計・背景 | コードを読んでも分からない「なぜこうしたか」を補足する |
| Testing | 検証方法 | レビュアーが再現できる手順を具体的に記述する |
| Impact / Risks | 影響・リスク | 副作用・破壊的変更・運用上の注意を記述する |
| Related Issues | チケット連携 | `Closes #番号` で自動クローズ、`Relates to #番号` で参照のみ |

### ステップ4: ユーザーへの提示と承認

以下の3点をセットで提示する:

```
## PR Summary
- タイトル: [提案するPRタイトル]
- ベースブランチ: main（または対象ブランチ）
- 変更コミット数: N件 / 変更ファイル数: M件

## PR Description Proposal
（ステップ3で生成した説明文全文）

## Need Approval
上記の内容でPRを作成してよいですか？
修正が必要な場合は内容をお知らせください。
```

**承認プロセスのルール:**
- ユーザーの承認が得られるまで `gh pr create` を実行してはならない
- 修正指示があった場合は指示を反映して再提案し、再度承認を得る
- 承認されたら、提示した内容をそのまま使用してPRを作成する
- ドラフトPRにするかどうかはユーザーに確認する（デフォルトは通常PR）

### ステップ5: PRの作成

承認を得たら以下を実行する:

```bash
gh pr create \
  --title "PRタイトル" \
  --body "$(cat <<'EOF'
## Summary
...（承認済みの説明文）
EOF
)" \
  --base main
```

作成完了後、PR URLをユーザーに報告する。

**オプションの使い分け:**

| 状況 | 追加オプション |
|---|---|
| まだレビュー依頼前（作業中） | `--draft` |
| レビュアーを指定したい | `--reviewer ユーザー名` |
| ラベルを付けたい | `--label "bug"` など |
| 内容を事前確認したい | `--dry-run` |

## 使用例

### 例: バグ修正PR

1. `git log main...HEAD --oneline` でコミット一覧を確認する。
2. ブランチ名 `bugfix/456-validation-error` から `Fixes #456` を抽出する。
3. `git diff main...HEAD` で変更内容を分析し、Summary・Changes・Context を生成する。
4. ユーザーに提示して承認を得る。
5. `gh pr create --title "[バグ修正] バリデーションエラーを修正" --body "..."` を実行する。

### 例: 機能追加PR（ドラフト）

1. 変更履歴を収集し、WIP状態であることを確認する。
2. ユーザーにドラフトPRとして作成するか確認する。
3. 承認後、`gh pr create --draft --title "..."` を実行する。

## エラー対応

| 状況 | 対応 |
|---|---|
| `gh` コマンドが未認証の場合 | `gh auth login` での認証を案内する |
| ブランチがリモートにプッシュされていない場合 | `git push -u origin ブランチ名` を実行してから再試行する |
| ベースブランチが不明な場合 | ユーザーに確認する（main / develop / master など） |
| 同名のPRが既に存在する場合 | 既存PRのURLを報告し、更新か新規作成かを確認する |

## セキュリティチェック

PR作成前に以下を確認する:

- [ ] APIキー・パスワード・トークンなどのシークレットがコードに含まれていない
- [ ] `.env` ファイルや認証情報ファイルが変更に含まれていない
- [ ] 機密情報を含むコメントや設定値がないか

シークレットが検出された場合はPR作成を中断し、ユーザーに警告する。

## ベストプラクティス

- **全履歴を見る**: 最新コミットだけでなく `git diff main...HEAD` で全変更を確認する
- **「なぜ」を書く**: 実装の理由・背景・設計判断をContextに必ず記述する
- **小さくまとめる**: 複数の目的が混在する場合、PRを分割することを提案する
- **レビュアー目線で書く**: テスト手順はレビュアーが再現できる粒度で具体的に書く
- **承認を省略しない**: どんな状況でもユーザー承認なしにPRを作成しない
